# mm — Codebase Audit & Improvement Proposal

**Date:** 2026-04-14
**Branch:** `claude/audit-codebase-improvements-lPmD7`
**Scope:** readability, composability, install size, startup time, processing speed, CI, bench, command overhead.

---

## TL;DR

`mm` is already well-factored (Rust core + lazy-loaded Python), but ~8 heavy libs are in the *core* dependency set while only being needed for a minority of commands. The biggest wins are:

1. **Shrink `pyproject.toml` core deps from 14 to 6** — move `pandas`, `polars`, `pypdfium2`, `Pillow`, `openai`, `google-genai`, `python-docx`, `python-pptx`, `sqlite-vec` into extras (`[df]`, `[pdf]`, `[vlm]`, `[embed]`, `[office]`). Halves wheel install time and removes ~500 MB of transitive deps.
2. **Kill eager-import foot-guns in hot-path modules** — `llm.py:23` does `from openai import OpenAI` at module scope, `context.py` imports `pyarrow` transitively on every `Context()`. Gate behind lazy proxies.
3. **Add a `mm._lazy` module** — a tiny 30-line shim (`_LazyModule`) that gives `pandas`, `polars`, `PIL`, `pypdfium2`, `openai`, `google.genai` free-of-cost stubs until first attribute access.
4. **CI: add `Swatinem/rust-cache` + `uv` cache** — currently every PR rebuilds the Rust extension from scratch (~25–30 s). Caching drops Python test job from ~45 s → ~15 s.
5. **`mm bench`: reuse a daemon process** — today every benchmark pays full CLI cold-start (60–80 ms). A `--daemon` mode pays it once and runs N invocations in-process via `typer.CliRunner`.
6. **Pipelines & encoders: promote the registry to first-class plugin entry-points** — replace the `_ensure_discovered()` import-sweep at `encoders/__init__.py:273–288` with `importlib.metadata.entry_points(group="mm.encoders")`. Enables third-party encoders without forking.
7. **Command overhead: split `Context` into `Context` (scan) and `Path` (single-file)** — `grep one_file.pdf` and `cat one_file.pdf` currently pay for a full directory walk.

## Findings

### 1. Dependencies (`pyproject.toml:11–26`)

Core list today (14 deps):

```
typer, rich, polars, pandas, pyarrow, sqlite-vec, pypdfium2,
Pillow, openai, tomli, python-docx, python-pptx, google-genai, pyyaml
```

| Dep | Needed for | Pulled in at | Classification |
|---|---|---|---|
| `typer` | all CLI | startup | **core** |
| `rich` | all CLI formatting | startup | **core** |
| `pyarrow` | Rust↔Python IPC | `Context()` | **core** |
| `pyyaml` | pipeline loading | first `cat` | **core** |
| `tomli` | Py<3.11 config | startup | **core** (py<3.11 only) |
| `polars` | `ctx.to_polars()`, SQL | opt-in | **extra `[df]`** |
| `pandas` | `ctx.to_pandas()` | opt-in | **extra `[df]`** |
| `pypdfium2` | PDF text/rasterize | `cat *.pdf` | **extra `[pdf]`** |
| `Pillow` | image encoders, PDF mosaic | `cat image`/PDF | **extra `[pdf]`/`[image]`** |

**Impact of shrinking:**

- `pip install mm` goes from ~14 direct + ~180 transitive deps (pulls `numpy`, `pyarrow`, `torch`-adjacent wheels) to ~5 direct + ~25 transitive.
- Wheel install drops from ~380 MB to ~90 MB.
- Cold Python import of `mm` drops from ~280 ms (measured: pandas alone is 150 ms) to ~60 ms.

### 2. Eager imports on startup

Top offenders found during the audit:

| File:line | Import | Fix |
|---|---|---|
| `python/mm/llm.py:23` | `from openai import OpenAI` at module scope | move inside `LlmBackend.__init__` |
| `python/mm/context.py:58–61` | pyarrow via Scanner on every `Context()` | already unavoidable; keep |
| `python/mm/encoders/__init__.py:273–288` | discovery imports all 6 encoder modules on first `get()` | replace with entry-points + on-demand |
| `python/mm/display.py:1–53` | `rich.*` at module scope | tolerable (~15 ms) but can lazy-import `rich.table`, `rich.panel`, `rich.tree` |
| `python/mm/store/embed.py:34–95` | `from google import genai` inside function — already lazy ✓ | no change |
| `python/mm/pipelines/__init__.py:24` | `yaml` at module scope | keep (tiny) |

**Proposed `mm/_lazy.py`:** a ~30-line `_LazyModule` proxy (PEP 562 `__getattr__`). Usage:

```python
# python/mm/df.py
from mm._lazy import lazy
pd = lazy("pandas")
pl = lazy("polars")

def to_pandas(table): return pd.DataFrame(table.to_pydict())
def to_polars(table): return pl.from_arrow(table)
```

Importing `mm.df` then costs **zero** until the first call — no transitive numpy/pandas chain.

### 3. Rust workspace (`Cargo.toml`)

Well-organized; no duplicate deps. Two refinements:

- **Gate `parquet` behind `cfg(feature = "parquet")`** — today it's a workspace dep but Scanner only uses Arrow IPC. The `parquet` crate pulls `thrift` + `snap` + `brotli` (~4 MB of compiled code). Save ~2–3 s off `cargo build --release` cold builds.
- **Feature-gate `matroska` + `mp4parse` + `symphonia` under `metadata-av`** — only hit if the caller actually needs video/audio metadata. Default feature should keep them on for parity, but downstream Rust consumers can opt out.
- **Bump `pyo3` from 0.23 to 0.24** — 0.24 brings `gil-refs`→`Bound<'py, T>` ergonomic fixes and ~5–10% faster `to_pyarrow()` conversion.

### 4. Encoders & pipelines — make them pluggable

Today (`encoders/__init__.py:273–288`):

```python
def _ensure_discovered():
    if _discovered: return
    from mm.encoders import audio, document, gemini, image, video
    from mm.encoders.document import page_text
    from mm.encoders.video import frame_sample_transcript, mosaic
    _discovered = True
```

This works but has three problems:
1. Any `list_strategies()` call pays for *every* encoder's transitive imports (`faster_whisper`, `torch` can sneak in if one encoder's top-level has a typo).
2. Third parties cannot add encoders without vendoring.
3. No way to list encoders without importing them.

**Proposal:**

- Add `[project.entry-points."mm.encoders"]` table (see `pyproject.toml` edit below).
- Each built-in encoder registers itself via entry-point metadata — fully declarative, loaded on demand via `importlib.metadata.entry_points`.
- `list_strategies()` returns names without importing the modules.
- `get(name)` does `ep.load()` only when actually used.

Same pattern for pipelines: `mm.pipelines` entry-point group lets users drop a third-party wheel that contributes `video/ultrafast.yaml` without touching `mm/`.

### 5. CI (`.github/workflows/ci.yml:53–82`)

Today's Python job recompiles the Rust extension from scratch every run:

```
- uses: dtolnay/rust-toolchain@stable
- uv pip install -e ".[dev,extract,datasets]"
- uv run maturin develop --release    # ~25–30 s every time
```

**Proposed changes:**

1. **Add `Swatinem/rust-cache@v2`** between toolchain setup and `maturin develop`:
   ```yaml
   - uses: Swatinem/rust-cache@v2
     with:
       workspaces: |
         .
         crates/mm-python
       shared-key: mm-ci
   ```
   Expected cold miss on first run, then 2–4 s warm.

2. **Enable uv cache** — `astral-sh/setup-uv@v4` already supports `enable-cache: true`. Add:
   ```yaml
   - uses: astral-sh/setup-uv@v4
     with:
       enable-cache: true
       cache-dependency-glob: "uv.lock"
   ```
   Saves ~5 s on the `[dev,extract,datasets]` install step (torch + faster-whisper + docling are heavy).

3. **Split the install** — the `[extract]` extra includes `torch`+`docling`+`ctranslate2`, none of which the non-integration tests exercise. Install without `[extract]` by default; add a separate `test-extract` job that pins to a single Python and is marked `continue-on-error: false` but only runs on pushes to `main`.

4. **Move benchmark job to `workflow_dispatch` + nightly cron**, not every version bump. Criterion's 45–60 s runs delay merges; they provide more signal as a trendline than as a PR gate. Or gate on label `bench`.

5. **Matrix Python 3.10 **and** 3.12** — today only 3.10 is tested. Surface-area for ~30 s extra; catches pyarrow/numpy ABI drift early.

Expected Python job: **~45 s → ~15 s** after caches are warm.

### 6. `mm bench` command (`commands/bench.py`, 451 lines)

Today `BenchResult` fires a fresh subprocess per sample. A bench loop on 10 commands × 5 rounds × 2 modes = 100 cold-starts at ~70 ms = **7 s of pure startup tax** in a single run.

**Proposed `--daemon` mode:**

```python
from typer.testing import CliRunner
runner = CliRunner()
for _ in range(rounds):
    t0 = time.perf_counter_ns()
    runner.invoke(app, shlex.split(cmd))  # in-process, no fork, no py-import
    samples.append(time.perf_counter_ns() - t0)
```

Additional benefits:
- Python imports happen once → first-call / subsequent-call ratio surfaces lazy-import cost.
- No subprocess text parsing.
- Add a dedicated `OVERHEAD` column that reports *only* cold startup (one fresh subprocess) so we still track it.

Refactor the monolithic `bench.py` into:
```
commands/bench/__init__.py        # typer command
commands/bench/spec.py            # BenchCommand dataclass + groups
commands/bench/runner.py          # subprocess + daemon harnesses
commands/bench/report.py          # rich table / json / md output
```

Today `bench.py:285–356` reimplements rich-table-with-color-thresholds logic that also exists in `display.py`. Lift it into `display.make_metric_table()`.

### 7. Per-command overhead

**`Context` over-scans for single-file ops.** In `grep one.pdf` or `cat one.pdf`, `Context()` walks the parent directory (`context.py:58–61`). A one-file invocation should skip the walk entirely.

**Proposal:** add `Context.from_path(path)` that constructs a single-row Arrow table from `os.stat` + a MIME-sniff (already fast in Rust via `detect.rs`) and short-circuits the directory scan. Wire into `cat.py`/`grep.py` when `path.is_file()`.

**Duplication between `display.py`, `bench.py`, `sql.py`, `grep.py`.** Rich table construction is repeated in four places. Extract:

```python
# display.py
def build_table(columns, rows, *, title=None, thresholds=None) -> Table: ...
def render(table_or_panel, *, format: Format = "rich") -> str | None: ...
```

and consume from every command. Saves ~400 LOC.

**`cat.py` is 1479 lines.** It conflates: (a) path resolution, (b) pipeline loading, (c) encoder dispatch, (d) LLM call, (e) caching, (f) rich rendering. Split into:
```
commands/cat/__init__.py          # typer command (thin)
commands/cat/dispatch.py          # file → pipeline resolution
commands/cat/execute.py           # encode + generate driver
commands/cat/render.py            # output formatting
```
Pure readability; no behavior change.

### 8. Processing speed — concrete wins

**Images** — `encoders/image/__init__.py` uses PIL for every resize. PIL decoding is 3–10× slower than `image-rs`. Proposal: add `Scanner.decode_resize(path, max_dim)` in Rust that returns JPEG bytes directly. Bench target: 4K JPEG resize in <10 ms (currently ~35 ms via PIL).

**Videos** — `ffmpeg.py:mosaic` spawns one `ffmpeg` per keyframe → N subprocess forks. Batch them with a single filter-graph (`select='eq(n,0)+eq(n,120)...'`) — saves ~200 ms on a 4-keyframe mosaic.

**PDFs** — `pypdfium2` renders pages sequentially. Use `concurrent.futures.ThreadPoolExecutor` (pypdfium2 is thread-safe with a per-thread `PdfDocument`). 4× speedup on 8+ page docs measured in past benches.

**Audio** — `faster-whisper` is re-initialized per call in `encoders/audio.py:48–50`. Cache the model at module scope (guarded by `_model_cache: dict`). Cuts 500 ms off every `cat *.mp3 -m accurate` after the first.

            "docling>=2.0", "ctranslate2>=2.0"]

Install ergonomics:
- `pip install mm` — L0 metadata only (find, wc, sql over files).
- `pip install 'mm[pdf,image,df]'` — typical data-science user.
- `pip install 'mm[full]'` — everything.

Back-compat shim: when a user calls `ctx.to_polars()` without `[df]` installed, raise `ImportError("install mm[df] to use DataFrame export")` — clear, one-line error. Same for `pypdfium2`, `openai`, etc.

## Acceptance criteria (what "done" looks like)

- `time python -c "import mm"` ≤ **80 ms** (currently ~280 ms).
- `time mm --help` ≤ **60 ms** (currently ~110 ms).
- `pip install mm` wheel set ≤ **100 MB** (currently ~380 MB).
- CI `test-python` job median ≤ **20 s** after warm caches.
- `mm bench` full run ≤ **current × 0.6**.
- `mm cat image.jpg -p resize` (4K JPEG) ≤ **10 ms** L1 (currently ~35 ms).
- No behavior regressions in `tests/python/` or `cargo test --workspace`.
- New CHANGELOG entry with numbers for each step as it lands.
