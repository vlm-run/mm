Strategies for encoding binary files with mm

Transforming images, video, audio, and documents into VLM-ready messages for different use cases.

June 2026 12 min read images video audio documents

Vision Language Models (VLMs) reason over visual inputs but are fundamentally bound by a finite context window. The 256K–1M token budget that feels generous for text proves insufficient once dense visual modalities dominate: capacity is bounded not by semantic complexity but by raw token consumption.

Information density is modality-dependent: a 10-second video clip carries more context than 10 seconds of audio, which carries more than 10 seconds of text.

Forwarding raw binary payloads directly to the inference endpoint is fundamentally incompatible with these constraints. Consider the extremes: a 4K image, a 3-hour 12GB video, or a scanned PDF lacking a text layer. Each incurs prohibitive token overhead at the model layer, so providers downsample internally: images to 480×480 or video to 5 FPS at 320p prior to tokenization.

For images, this is often tolerable, since downsampling incurs little loss of semantic fidelity. But it remains architecturally wasteful: transmitting a 20MB 4K image to the inference endpoint, only to have it immediately rescaled to a 32KB thumbnail, is wasted bandwidth and added latency. This motivated our investigation into modality-aware encoding applied before the inference call.

Encoders

We formalize each such strategy as an encoder: a local, deterministic transform from a binary file to one or more messages, the OpenAI-compatible content parts (text, image_url, input_audio, video_url) that any chat/completions endpoint accepts. Each is identified by name (e.g., resize, mosaic, transcribe), bound to a single media kind, and answers one question for its modality: what is the most compact representation that still preserves content fidelity for the model layer?

The transform runs on-device before a single token is billed and generates transmittable message segments rather than a monolithic payload, so long or large inputs decompose naturally into smaller chunks transferred over parallel calls.

A file is encoded into OpenAI-compatible messages, then sent to a VLM for generation.

The diagram above conveys what happens end to end: a file reaches the encoder, which applies modality-aware transformation, e.g., resize a photograph, transcode a clip, transcribe speech, or render a document page, producing messages a VLM can process.

A key advantage of this design is that the number of generated messages adapts to the content. A small binary document encodes into a single message, while a large or densely packed one fans out into many messages, each a self-contained part sized to the model’s context window, allowing for comprehensive understanding.

Every aspect of the input is read and processed in parallel before being merged into a final output.

Image Encoders

Images are the simplest case: one file becomes a message that the model reads in a single pass. mm provides two strategies, each suited to different image characteristics.

resize
default

Fits the image into a bounding box (default 1024 px) while preserving aspect ratio and EXIF orientation. Outputs a single image message that the model reads in one pass. The right choice for most images.

max_width1024 px, maximum dimension (width or height) 
%%{init: {'look': 'neo', 'theme': 'neutral'}}%%
graph LR
  img("🖼️ image")
  subgraph encode ["Encode"]
    exif("Apply EXIF\norientation")
    resize("Fit to ≤ 1024 px\nbounding box")
    b64("Base64 encode")
  end
  subgraph message ["Message"]
    msg("image_url part")
  end
  img --> exif --> resize --> b64 --> msg
  style encode rx:10px,ry:10px
  style message rx:10px,ry:10px
resize: EXIF correction → fit to bounding box → single image message

Use resize for portraits, product photos, screenshots, logos, and any image where content fits comfortably at 1024 px. For most real-world images, this is the right choice, the fastest path, and the cleanest input to a VLM.

tile
high resolution fine detail

Sends a resized overview (global context) plus NxM tile crops from the original (fine detail), all in a single message. Gives VLMs both the big picture and the ability to read small text or inspect fine-grained features at full resolution. Falls back to overview-only when the image fits in a single tile.

max_width1024 px, tile size, and overview bounding box

  subgraph encode ["Encode"]

    overview("Resize overview\nglobal context")
    tiles("Crop NxM tiles\nfine detail")
  end
  subgraph message ["Message"]
    msg("overview image\n+ N tile crops")
  end
  img --> exif
  exif --> overview --> msg
  exif --> tiles --> msg

tile: overview plus NxM crops in one message, global context and fine detail at once
🔍
Tile vs Resize

Use tile for technical diagrams, architectural drawings, scanned documents where text must be recognized, maps, and medical imaging, anywhere that important detail would be lost at 1024 px. Use resize for everything else: photos, icons, screenshots, illustrations.

bash 
# Default: resize (fast pipeline)
mm cat photo.jpg

# High-resolution diagram: tile strategy
mm cat blueprint.png -p tile

# Smaller tile for very large images
mm cat satellite.tiff -p tile --encode.strategy_opts max_width=768

# Custom prompt on accurate mode
mm cat diagram.png -p tile -m accurate --prompt "Extract all labels and measurements"

Video Encoders

Video is the richest and most challenging media type to encode. A 90-minute lecture can't fit in one message. A 30-second product clip may need both its frames and audio characteristics to be understood. mm provides 19 video encoding strategies across four fundamental approaches: frame-based, time-chunk-based, native clip, and audio-only.

Video encoder decision tree: start top-left and follow the constraints

In the diagram above, if the model is video-native, then frame extraction is skipped and the clip is sent as-is. On the other branch, given that most models aren’t video-native, the video is reduced to images and text. From there, three constraints decide the strategy. Duration sets the sampling budget: a 10-minute clip can be sampled densely, a 90-minute lecture can't. Audio decides whether the encoder also transcribes the clip and sends the text alongside the frames. Content type decides how frames are chosen: uniformly, per scene, or as a single adaptive overview. Each leaf of the tree is one of the encoders below, and the rest of this section covers them in that order, starting with mosaic.

mosaic
default

Extracts representative frames using scene detection (falling back to uniform sampling) and tiles them into 4×4 grids. The result is a compact visual summary that fits in a small number of messages, enough for a VLM to understand what happens in a clip without processing every frame.

  subgraph encode ["Encode"]
    detect("Scene detection\nor uniform sampling")
    frames("Extract up to\n128 frames")
    tile("Tile into 4×4\nmosaic grids")
  end
  subgraph message ["Message"]
    msg("N mosaic grid images")
  end
  video --> detect --> frames --> tile --> msg
  style encode rx:10px,ry:10px
  style message rx:10px,ry:10px
mosaic: scene-aware sampling → tiled grids → compact visual summary
bash 
# Default
mm cat clip.mp4

# With transcript
mm cat clip.mp4 -p mosaic-w-transcript

# Denser 3×3 grid
mm cat clip.mp4 -p mosaic --encode.strategy_opts tile_cols=3 --encode.strategy_opts tile_rows=3

# More frames for long content
mm cat lecture.mp4 -p mosaic --encode.strategy_opts num_frames=256
frames / frames-w-transcript

Extracts frames at a fixed rate (default 1 fps) and batches them into messages of up to 16 frames each, with a time-range header per batch. The -w-transcript variant prepends a Whisper transcript as the first message, giving the VLM both visual context and spoken word.

    batch("Batch ≤ 16\nframes / message")
  end
  subgraph message ["Message"]
    t("1: transcript text")
    f("2..N: frame batches\nwith time headers")
  end
  video --> audio --> whisper --> t
  video --> fps --> batch --> f

  style message rx:10px,ry:10px
frames-w-transcript: audio and frames extracted in parallel → transcript first, then batched frame messages
shots / shots-w-transcript
shot-boundary aware

Shot detection captures scene boundaries and yields one message per shot, with up to 8 representative frames each. Unlike frames, which are sampled uniformly, shots encoding is content-aware: each message maps to a semantically distinct scene. Ideal for films, documentaries, news broadcasts, and presentations with clear slide transitions.

shots: content-aware, each message is one semantically distinct scene
chunks
long-form temporal detail

Divides a video into overlapping time windows (default 60 s, 5 s overlap) and extracts 16 frames per window. Each chunk becomes a separate message with a time-range header. Unlike a single mosaic that compresses everything into a few grids, chunks preserve temporal granularity: the VLM understands what happened and when.

chunk_duration60 s, length of each time window overlap5 s, overlap between consecutive windows frames_per_chunk16, frames extracted per chunk

bash 
# 60 s windows, 5 s overlap, 16 frames per window
mm cat lecture.mp4 -p chunks

# Longer windows for slower-paced content
mm cat lecture.mp4 -p chunks --encode.strategy_opts chunk_duration=90 --encode.strategy_opts overlap=10

# Accurate mode: the model describes each window
mm cat lecture.mp4 -p chunks -m accurate

# Preview the resolved pipeline without running it
mm cat lecture.mp4 -p chunks -m accurate --dry-run

Other video strategies

keyframes
Extract I-frames directly from the video bitstream, no seeking loop, codec-aware. Fast way to get the natural edit points in a video.
bitstream
shot-mosaic
Shot boundaries + one tiled mosaic grid per shot. Compact representation that still preserves the structure of scene changes.
shot-aware
summary
Adaptive N-frame overview (default 12) spread across scene boundaries. Best for very long videos where a compact bird's-eye view is enough.
compact
clips
Send the video as base64-encoded clips for models that accept native video input. Videos ≤ 120 s sent whole; longer split into overlapping segments.

transcript
Whisper transcript only, no visual frames extracted. For podcasts, lectures, or any content where spoken word is all that matters.
encode-only
captions
Extract embedded subtitle tracks (SRT/VTT/SSA). Falls back to Whisper when no subtitle stream is found in the container.
encode-only
gemini
Send the entire video as a Gemini native Part. No frame extraction; the model receives the raw video bytes directly.
Gemini only
gemini-chunked
Same as gemini but splits long videos into overlapping segments first. For Gemini models with length constraints.

Critical distinction

transcribe is encode-only: it runs Whisper locally and returns a timestamped transcript as plain text, with no model in the loop. base64 and gemini generate model-native messages.

Encoder What it sends Output
transcribe Whisper transcript as text timestamped transcript
base64 Raw audio bytes to LLM description
gemini Audio as Gemini native Part description

Both base64 and gemini handle long files automatically: recordings longer than 120 seconds are split into overlapping segments and processed in parallel, each yielding a separate message.

bash 
# Default: Whisper transcript, no LLM
mm cat meeting.mp3

# Model description, accurate (full content analysis)
mm cat meeting.mp3 -p base64 -m accurate

# Faster transcription on Apple Silicon
mm cat meeting.mp3 -p transcribe --encode.backend mlx

# Gemini native audio understanding
mm --profile gemini cat meeting.mp3 -p gemini -m accurate

mm cat meeting.mp3 -p transcribe --encode.strategy_opts audio_speed=1.0

Document Encoders

Documents sit in an interesting middle ground: most PDFs have a text layer, but many don't, including scanned pages, image-heavy forms, and complex visual layouts. mm provides four strategies depending on what is required from the document.

Document encoder selection, driven by whether the PDF has a text layer and whether visual layout matters
page-text
Extract text per page from PDF, DOCX, and PPTX, no rendering, no images. Batches up to 128 pages per message. Fast and token-efficient. Default for fast mode.
default
rasterize
Render PDF pages as JPEG images, up to 4 pages per message with a page-range header. Preserves full visual layout: tables, charts, handwriting, stamps, signatures.
visual layout
rasterize-text
Rasterize pages and interleave the extracted text beneath each image. The VLM sees the visual layout and has an OCR-quality text fallback for every page.
hybrid
gemini
Send the document as a Gemini native Part, no text extraction or rendering. For Gemini models that accept PDF files directly.

Writing Custom Encoders

The encoder registry is open, allowing anyone to integrate custom encoders. Drop a .py file in ~/.config/mm/encoders/ and mm automatically discovers it at startup. Two authoring styles are supported.

Class-based

python 
from pathlib import Path
from typing import Any, Iterable
from mm.encoders import register
from mm.encoders.base import Encoder, Message

class VideoGrayscale(Encoder):
    """Encode video as grayscale frame thumbnails."""
    name = "grayscale"
    kind = "video"

    def encode(self, path: Path, **kwargs: Any) -> Iterable[Message]:
        max_width: int = kwargs.get("max_width", 512)
        # extract frames, convert to grayscale, base64-encode
        yield {"role": "user", "content": [
            {"type": "text", "text": f"Grayscale frames from {path.name}"}
        ]}

register(VideoGrayscale())

Function-based

python 
from pathlib import Path
from mm.encoders import register_encoder

@register_encoder(name="ocr-fallback", kind="image")
def ocr_fallback(path: Path, **kw):
    """Run pytesseract OCR and return extracted text."""
    import pytesseract
    text = pytesseract.image_to_string(str(path))
    yield {"role": "user", "content": [{"type": "text", "text": text}]}
🔌
Encoder discovery

mm picks up any .py file in ~/.config/mm/encoders/ at startup. Run mm cat --list-encoders to confirm your encoder is registered and available via -p <name>.
